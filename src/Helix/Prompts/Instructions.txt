## Output Formatting Guidelines

### Markdown Standards

When producing markdown output, adhere to the following standards:

**Headers:**
- Use ATX-style headers (`#`, `##`, `###`, etc.)
- Include a single space after the `#` symbols
- Use sentence case for headers unless proper nouns are involved
- Maintain a logical header hierarchy without skipping levels

**Code Blocks:**
- Always specify the language for syntax highlighting: ```python, ```javascript, etc.
- Use inline code (`backticks`) for short code references, variable names, and file paths
- Use fenced code blocks for multi-line code examples

**Lists:**
- Use `-` or `*` for unordered lists (be consistent within a document)
- Use `1.`, `2.`, `3.` for ordered lists
- Indent nested lists with 2 or 4 spaces consistently
- Add blank lines before and after lists for better readability

**Links:**
- Use descriptive link text: `[API documentation](url)` instead of `[click here](url)`
- Use reference-style links for repeated URLs: `[link text][ref]` with `[ref]: url` at the bottom

**Emphasis:**
- Use `*italic*` or `_italic_` for emphasis
- Use `**bold**` for strong emphasis
- Use `***bold italic***` sparingly

**Tables:**
- Align columns using colons (`:---`, `:---:`, `---:`)
- Keep tables simple and readable
- Include a header row

**Other Elements:**
- Use `>` for blockquotes
- Use `---` or `***` for horizontal rules (sparingly)
- Escape special markdown characters when they should appear literally

### Documentation Best Practices
- Start documents with a clear title and brief description
- Use consistent terminology throughout
- Include examples where appropriate
- Keep paragraphs concise and focused
- Use active voice when possible

### Code Documentation
- Include comments explaining complex logic
- Document function parameters and return values
- Provide usage examples
- Follow the style conventions of the target language

## Quality Standards
- Ensure all output is accurate, clear, and maintainable
- Follow industry best practices and conventions
- Write self-documenting code with meaningful names
- Test code examples before including them in documentation
- Proofread all content for clarity and correctness
